# NexusSite-AI 官方 PRD（产品需求文档）

- **文档版本**: v1.0  
- **产品代号**: NexusSite-AI  
- **撰写日期**: 2026-03-31  
- **目标读者**: 产品 / 设计 / 研发 / 测试 / 运维 / 交付  
- **文档范围**: 本 PRD 覆盖 NexusSite-AI 的核心定位、功能模块、技术规范与验收标准，面向本地 Docker 工作流的一体化“对话即交付”官网生成系统。

---

## 1. 背景与问题定义

### 1.1 背景
独立开发者、创业团队与中小企业常常需要在极短时间内搭建一个“可上线、可迭代、具备现代审美”的官网。传统流程（需求、设计、开发、测试）链路长、成本高、沟通损耗大，且交付质量难以稳定。

### 1.2 目标问题
- **需求表达不标准**：用户多以自然语言描述，缺少可执行的规格说明。
- **设计与开发割裂**：设计资产（tokens/组件）无法结构化沉淀，UI 风格难统一。
- **生成结果不可用**：代码不通过构建、类型错误、组件不可维护。
- **缺少质量闭环**：无法自动发现并修复构建/视觉问题，导致“生成即失败”。

---

## 2. 产品定位与愿景

### 2.1 产品定位
**全链路自动化 AI 官网工作室**：以本地 Docker 工作流为载体，通过多智能体协作（MAS），在一个统一的可视化控制中心内完成从需求到交付的闭环。

### 2.2 产品愿景
让 **“对话即交付”** 成为现实：把专业软件公司的研发流程（需求 → 设计 → 开发 → 测试）压缩为**可复现、可自愈、可交付**的本地交付流水线。

---

## 3. 目标用户与典型场景

### 3.1 目标用户
- **独立开发者/小团队**：需要快速生成产品官网、营销页、文档入口。
- **初创公司**：需要可迭代的官网原型用于融资/冷启动。
- **增长/市场**：希望快速试验落地页方案并迭代文案结构与 SEO。

### 3.2 典型使用场景（User Stories）
- **US-01**：作为用户，我输入产品简介与目标客户，系统在 5 分钟内输出可运行官网原型，并能实时预览。
- **US-02**：作为用户，我希望系统产出包含 sitemap/SEO 基础结构的官网，以便后续上线。
- **US-03**：作为用户，我希望系统在构建失败时自动诊断并修复，直到 build 成功。
- **US-04**：作为用户，我希望一键导出源代码或构建产物，便于部署或交付给开发团队。

## 4. 产品目标与非目标

### 4.1 产品目标（Goals）
- **G1**：自然语言需求 → 标准化 PRD → 设计 tokens → 可运行 Next.js 官网原型。
- **G2**：提供可视化控制中心：对话、智能体状态、工作流图、实时预览。
- **G3**：提供自愈闭环：构建与视觉验收失败时自动回路修复。
- **G4**：支持导出交付物：`dist`（或等价构建产物）与源代码压缩包。

### 4.2 非目标（Non-goals）
- **NG1**：不覆盖复杂业务系统（如支付、完整后台管理、复杂权限体系）。
- **NG2**：不以云端托管为前提（v1 以本地 Docker 运行优先）。
- **NG3**：不追求“一次生成即完美”，而追求“可迭代、可自愈、可交付”。

### 4.3 范围边界与复杂度上限（Complexity Limits）
#### 4.3.1 支持范围（v1 硬边界）
- **网站类型**：仅支持 **Marketing Sites（展示型官网/落地页）**。
- **不包含能力**（v1 明确不做）：
  - 登录/注册/用户体系、权限体系
  - 数据库写操作与完整 CRUD（订单、库存、购物车等）
  - 支付/交易链路、复杂工作流与状态机业务

#### 4.3.2 规模指标（建议作为产品保护阈值）
- **页数**：1–5 页
- **单页原子组件数**：≤ 15（不含 Shadcn UI 基础组件的内部拆分）

#### 4.3.3 超限处理策略（例如 50 页需求）
- **默认行为**：在 PM 阶段提示项目规模过大，**拒绝一次性生成**。
- **替代方案**：引导用户拆分为多个子任务，或先生成 **核心 3–5 个页面模板**（信息架构、组件体系、视觉基调）再分批扩展。

## 5. 核心功能模块（Features）

### 5.1 智能体专家团（The Elite Agents）
系统以多智能体协作实现端到端交付。每个智能体有明确输入/输出契约（Artifacts），并受编排层约束。

#### 5.1.1 Strategist PM（策略产品经理）
- **职责**：
  - 将用户自然语言转化为**标准化 PRD**（结构化章节、功能清单、页面结构）。
  - 产出 **SEO 策略**：关键词方向、页面信息架构、标题/描述建议。
  - 产出 **Sitemap 规划**：站点路由树、优先级、页面用途说明。
- **输入**：用户对话需求、竞品/风格偏好（可选）、行业信息（可选）。
- **输出（Artifacts）**：
  - `PRD.md`
  - `sitemap.json`（或等价结构化路由描述）
  - `seo-brief.md`

#### 5.1.2 UI/UX Designer（设计师）
- **职责**：
  - 生成 **Design Tokens**（色彩、字体、间距、圆角、阴影、动效基线）。
  - 输出组件布局逻辑与页面栅格规则，确保整体符合现代 SaaS 审美（Linear/Stripe 风格）。
- **输入**：PRD、Sitemap、用户品牌信息（可选）、参考风格（可选）。
- **输出（Artifacts）**：
  - `design-tokens.json`
  - `ui-spec.md`（组件清单、布局规则、交互说明）

#### 5.1.3 Senior Engineer（高级工程师）
- **职责**：
  - 基于 PRD/设计 tokens 生成 **Next.js + Tailwind CSS + Shadcn UI** 项目代码。
  - 采用**原子化组件架构**（可维护、可复用、可扩展）。
  - 保证 TypeScript 类型正确、构建可通过。
- **输入**：PRD、Sitemap、Design Tokens、UI Spec。
- **输出（Artifacts）**：
  - 增量代码变更（patch 形式）
  - `README.md`（本地运行与构建说明）

#### 5.1.4 QA Guardian（质量守护）
- **职责**：
  - **逻辑验收（Build Check）**：执行 `npm run build`，分析错误日志并定位根因。
  - **视觉验收（VQA）**：根据页面截图/关键视图对齐 UI Spec，检查响应式与视觉一致性。
  - 将失败反馈结构化给 Engineer，触发自愈回路。
- **输入**：构建日志、运行时日志、页面截图、验收标准。
- **输出（Artifacts）**：
  - `qa-report.md`（失败原因、复现步骤、修复建议、通过证明）
  - `vqa-results.json`（关键视图与断点检查结果）

---

### 5.2 基础设施与环境（Infrastructure）

#### 5.2.1 Dual-Container Sandbox（双容器沙盒）
- **Orchestrator 容器**
  - 运行 LangGraph 与多智能体编排逻辑
  - 提供 FastAPI 服务与 WebSocket 日志流
- **Workspace 容器**
  - 运行 Next.js Dev Server
  - 通过 Docker 卷挂载实现代码实时热更新（Orchestrator 写入 patch → Workspace 即时生效）

#### 5.2.2 Interactive Control Center（交互控制中心）
一个极简主义 Web UI，用于承载核心交互与可观测性：
- **聊天窗口**：用户输入需求、追问与确认信息（以对话驱动）。
- **Agent 状态追踪器**：展示每个智能体状态、当前任务、产出物链接。
- **LangGraph 可视化**：显示执行节点、分支与回路、失败重试路径。
- **实时预览 Iframe**：嵌入 Workspace 的本地预览页面（支持刷新/切换路由）。

---

### 5.3 自愈闭环（Self-healing Loop）

#### 5.3.1 回路规则（必须）
系统必须支持如下循环，直到达成验收：
- **Coder → QA → Coder**
- 只有当：
  - `npm run build` 成功
  - 且视觉验收通过（响应式关键断点、核心页面视觉一致）

才允许进入**交付阶段**（export）。

#### 5.3.2 失败策略（建议）
- **构建失败**：优先修复 TypeScript、依赖、Next.js 约束问题。
- **视觉失败**：优先修复布局断点、字体/间距 tokens 映射、组件层级结构。
- **无限循环防护**：设置最大迭代次数与降级策略（例如先保证 build，再逐步提升视觉质量）。

## 6. 交互流程与工作流（High-level Flow）

### 6.1 主流程
1. 用户在控制中心输入自然语言需求  
2. Strategist PM 输出 PRD + Sitemap + SEO Brief  
3. UI/UX Designer 输出 Design Tokens + UI Spec  
4. Senior Engineer 以 patch 方式生成/更新代码并启动预览  
5. QA Guardian 执行 build 与视觉验收  
6. 若失败：回到 Engineer 修复（自愈回路）  
7. 若通过：进入交付导出（dist/源代码）

### 6.2 可观测性与产出物
- 控制中心应显示：
  - 当前阶段、耗时、产出文件列表
  - 实时日志（WebSocket）
  - 构建结果与 VQA 结果摘要

## 7. 技术规范（Technical Specs）

### 7.1 技术栈约束
- **协调层（编排）**：LangGraph（Python）
- **后端**：FastAPI + WebSocket（实时传输日志/状态）
- **前端（控制中心）**：Next.js + Tailwind + Lucide Icons
- **代码生成策略**：增量文件操作（Patching），禁止全量重写（除非初始化）
- **沙盒**：Docker（Node.js 20-slim）

### 7.2 关键工程约束
- **类型安全**：生成代码必须通过 TypeScript 编译与 Next.js 构建。
- **组件体系**：Shadcn UI 为基础组件层，业务层组件原子化拆分。
- **运行隔离**：Orchestrator 与 Workspace 职责分离，减少互相污染。
- **实时热更新**：通过卷挂载与 dev server 实时反馈生成结果。

### 7.3 模型与供应商策略（LLM Routing）
#### 7.3.1 默认模型分配（v1 推荐）
- **Strategist PM / UI/UX Designer / QA Guardian（含视觉）**：默认使用 **GPT-4o**（指令遵循与多模态稳定性优先）。
- **Senior Engineer**：默认使用 **Claude 3.5 Sonnet**（代码生成质量、长上下文与 Tailwind 熟练度优先）。

#### 7.3.2 统一路由与可切换（避免供应商锁定）
- **原则**：所有智能体调用统一经由 **LLM Router**，在配置层完成 `agent -> provider/model` 映射，不在业务代码中硬编码供应商。
- **配置**：在 `.env`（或等价配置文件）提供 `LLM_ROUTER_CONFIG`，支持通过 LiteLLM 等中间件进行供应商切换与降级。
- **降级策略（建议）**：
  - 供应商/模型不可用时，允许切换到同等能力的备选模型。
  - 对 QA/VQA 这类门禁环节，降级必须保证“可解释输出 + 可复现证据（日志/截图）”不丢失。

### 7.4 视觉验收（VQA）实现规范（v1 最小可行）
#### 7.4.1 多模态模型
- **默认**：使用 **GPT-4o（vision）** 进行截图理解与视觉一致性判定。

#### 7.4.2 截图触发策略（Event-driven）
为控制成本与时延，v1 仅在以下关键节点截图：
1. **全局骨架完成**（导航/布局/主要分区落位后）
2. **核心内容填充完成**（文案、图标、卡片、定价等主要内容齐备后）
3. **视觉缺陷修复后回归**（Engineer 声称修复某个视觉问题后）

#### 7.4.3 响应式断点（v1 强制）
- **Mobile**：375px
- **Desktop**：1440px
- （可选扩展）Tablet：768px（不作为 v1 强制门禁）

### 7.5 增量变更策略（文件级原子覆盖）
#### 7.5.1 变更粒度
- **禁止**：让 LLM 直接生成标准 diff/patch（易格式错误、难回放、难定位）。
- **采用**：**文件级原子覆盖（Atomic File Rewrite）**。即每次修改以“完整文件内容”为单位写入（例如修改 `Hero.tsx` 则覆盖写入完整 `Hero.tsx`）。

#### 7.5.2 依赖与联动
- 组件接口变更导致的连锁错误，通过 **TypeScript 编译 / Next.js build** 作为硬反馈暴露。
- QA Guardian 必须将报错日志结构化反馈给 Engineer，并驱动最小范围的文件覆盖修复。

#### 7.5.3 允许全量重写的触发条件（受控）
仅在以下场景允许“全量重写/大范围重构”：
- **项目初始化**（首次生成工程）
- **核心全局配置重大变更**（例如 `tailwind.config.*`、`tsconfig.json`、Next.js 目录结构迁移等）

## 8. 验收标准（Acceptance Criteria）

### 8.1 必须满足（MUST）
1. **时效性**：用户输入需求后，**5 分钟内产出可运行的官网原型**（可在预览中访问核心页面）。
2. **构建可用**：生成的代码必须通过 TypeScript 编译，且 `npm run build` 成功。
3. **响应式**：生成 UI 必须是响应式（至少覆盖移动端/桌面端关键断点）。
4. **可交付导出**：用户可一键导出 **`dist`（或等价产物）** 或 **源代码压缩包**。

### 8.2 建议满足（SHOULD）
- Sitemap 与基础 SEO 字段完整（title/description/开放图）
- 控制中心可追溯每次迭代的产出与 QA 报告
- 自愈回路可解释（显示失败原因与修复步骤）

### 8.3 质量门禁与自愈循环（Release Gates）
#### 8.3.1 门禁顺序（强制）
- **Gate 1：类型/构建门禁**：必须通过 TypeScript 编译（`tsc`）与 `npm run build`。
- **Gate 2：视觉门禁（VQA）**：通过关键页面在 Mobile（375px）与 Desktop（1440px）的截图验收。
- 仅当 Gate 1 与 Gate 2 均通过，才允许进入“交付导出”阶段。

#### 8.3.2 自愈回路上限（防止死循环）
- **最大重试次数（强制）**：`Max retry = 3`
- 超过上限后系统必须：
  - 暂停自动修复
  - 在控制中心提示“需要人工介入”
  - 允许导出当前上下文（最新日志、失败截图、最近变更文件清单）

## 9. 里程碑（Milestones）

- **M1（可用原型）**：控制中心可对话，完成 PRD → tokens → 生成 Next.js 项目并可预览  
- **M2（自愈闭环）**：引入 QA Guardian，构建失败可自动回路修复至成功  
- **M3（视觉验收）**：引入截图与断点检查，VQA 失败可回路修复  
- **M4（交付导出）**：一键导出源代码与构建产物，生成交付清单

---

## 10. 风险与对策（Risks）

- **R1：生成质量不稳定**  
  - 对策：引入强约束产物契约（Artifacts），QA 门禁（build+VQA）作为交付前置条件
- **R2：自愈回路陷入重复失败**  
  - 对策：最大迭代次数、错误分类与优先级策略（先 build 后视觉）
- **R3：本地环境差异导致不可复现**  
  - 对策：Docker 固化 Node 版本与依赖安装流程；日志与产物完整保存
- **R4：增量 patch 导致代码漂移**  
  - 对策：关键文件锁定策略、模块边界约束、必要时触发“受控重构”节点

## 11. 附录：术语表（Glossary）

- **MAS**：Multi-Agent System，多智能体协作系统  
- **Artifacts**：工作流产出物（PRD、tokens、代码 patch、QA 报告等）  
- **VQA**：Visual QA，基于截图/断点的视觉验收  
- **Patching**：增量修改文件的方式，避免全量重写以提升可维护性与可追溯性
